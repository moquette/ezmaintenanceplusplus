# Restore defect B: TRIGGER IDENTIFIED AND REPRODUCED (2026-07-18)

Status: **trigger confirmed on real Kodi, twice, on the local macOS bench.**
No Fire TV or Apple TV was used. Supersedes the "TRIGGER UNIDENTIFIED" status in
`TASKS.md:67-90` and `docs/restore-defects-2026-07-18.md`.

## The trigger is in skin.estuary7, not in EZM++

`skin.estuary7/xml/Home.xml:9` (shipped 1.0.66, generated by
`estuary7/tools/skin_transforms.py:474-478`):

```xml
<onload condition="String.IsEmpty(Window(10000).Property(t7b_firstbuild_done))">AlarmClock(t7bbuild,RunScript(script.skinshortcuts,type=buildxml&amp;mainmenuID=9000&amp;group=mainmenu),00:15,silent)</onload>
```

On the first Home load of every boot, on BOTH platforms (this line carries no
`System.Platform` condition), the skin arms a 15-second alarm that runs a
skinshortcuts `buildxml`. When the menu is stale, skinshortcuts finishes with
`xbmc.executebuiltin("ReloadSkin()")`
(`script.skinshortcuts/resources/lib/skinshorcuts/xmlfunctions.py:123`), which
tears down the entire window stack. **Every open dialog dies, including EZM++'s
post-restore prompt.**

## Reproduction, run 2: the full defect

Bench: macOS, Kodi 21.3, virgin profile, skin.estuary7 1.0.66,
EZM++ 2026.07.17.7, `.ezm_buffer_prompt` marker armed, skinshortcuts addon_data
cleared to simulate restored (stale) menu data.

```text
17:50:20.052  started alarm with name: t7bbuild        <- skin arms the 15s timer
17:50:20.140  Window Init (DialogSelect.xml)           <- EZM++ prompt 1 of 2 opens
17:50:35.106  CPythonInvoker(5): type=buildxml         <- alarm fires, 15.05 s later
17:50:35.403  Window Deinit (Home.xml)                 <- ReloadSkin tears the stack down
17:50:35.405  Window Deinit (DialogSelect.xml)         <- THE PROMPT IS DESTROYED
```

The prompt was open **15.27 seconds** and was destroyed with no user input. The
`.ezm_buffer_prompt` marker was **consumed** on that boot, so the flow treated
the destroyed dialog as an answer and will not fire again.

## Reproduction, run 1: the trigger in isolation

Same bench, no EZM++ involved: alarm armed `17:43:51.988`, `buildxml`
`17:44:07.112`, second `load skin from ... skin.estuary7` at `17:44:07.434`.
**15.45 s.** Confirms the timing is a property of the skin alone.

## The post-restore asymmetry is real, and confirmed

An intermediate boot armed the alarm but produced **no** reload: skinshortcuts
had already built the menu, so `shouldwerun()` returned False and the build
returned early without calling `ReloadSkin()`. The reload only happens when the
menu is genuinely stale, which is exactly what a restore produces (extracted
`addon_data/script.skinshortcuts/` no longer matches the on-disk hash). That is
the whole normal-boot vs post-restore difference, with no platform component.

## CORRECTION to the recorded defect

The record says the **keyboard** is torn down mid-typing. On this reproduction
the **`DialogSelect` (prompt 1 of 2, "Device name") was destroyed before the
keyboard ever opened.** The keyboard is only reached if the user picks "Rename"
within 15 seconds.

This widens the defect rather than narrowing it. `dialog.select` returns `-1`
for both "user cancelled/back" and "dialog destroyed", exactly as
`_get_keyboard` conflates cancel with teardown. So
`prompt_devicename_after_restore` (`tools.py:421-461`) has the ambiguity **twice**:

1. `tools.py:433` - `idx = dialog.select(...)`, then `if idx != 0: return False`
2. `tools.py:441` - `entered = _get_keyboard(..., cancel=cur)`

Both must be fixed. A fix that only hardens `_get_keyboard` leaves the more
likely path (destroyed at the select stage) still broken.

## SEPARATE BUG, newly surfaced, not previously recorded

```text
17:51:54.427 error <general>: CPythonInvoker(2, .../script.ezmaintenanceplusplus/service.py): script didn't stop in 5 seconds
```

EZM++'s service does not honour Kodi's abort request within the 5-second
shutdown budget, so Kodi kills it. Reproduced on the bench; it is not related to
defect B and is not caused by the marker. Likely a `waitForAbort` /
`monitor.abortRequested()` gap in the maintenance loop, or a blocking call that
ignores the monitor. **Filed as its own item; do not bundle it into the B fix.**

## Bench recipe (repeatable)

1. Back up `~/Library/Application Support/Kodi` to `~/Kodi-bench-backups/`.
2. Wipe the profile, launch Kodi once to generate a virgin one, quit.
3. Install skin.estuary7 1.0.66 plus deps `script.skinshortcuts`,
   `script.image.resource.select`, `script.module.autocompletion`.
4. Install EZM++ from `dist/`.
5. Enable everything with Kodi CLOSED:
   `sqlite3 userdata/Database/Addons33.db "update installed set enabled=1;"`
   (manually copied addons land `enabled=0` and fail silently).
6. Set the skin by replacing the whole element, dropping `default="true"`:
   `<setting id="lookandfeel.skin">skin.estuary7</setting>`.
7. `<loglevel hide="false">1</loglevel>` in `userdata/advancedsettings.xml`.
8. Arm: `touch userdata/addon_data/script.ezmaintenanceplusplus/.ezm_buffer_prompt`
9. Make the menu stale: `rm -rf userdata/addon_data/script.skinshortcuts`
10. Launch, wait 90 s, quit. Read `~/Library/Logs/kodi.log` (NOT under the
    profile).
