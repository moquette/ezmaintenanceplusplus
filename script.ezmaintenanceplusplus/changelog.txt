2026.07.15.0
 - First-ever run now offers the same two-step tune-up a restore does: name this
   device, then size the video cache buffer for it. Exactly-once per install;
   boxes that already ran EZ Maintenance++ are not re-prompted by the update.

2026.07.15.1
 - Backup now confirms the final backup name (with its auto timestamp) before it
   starts, matching the restore confirm. Cancelling that prompt writes nothing.
   Applies to local, network, and Dropbox backups.

2026.07.15.2
 - Maintenance: new "Clear All" runs cache, packages, thumbnails, and (if any
   are found) the recently played channels in one action.
 - Maintenance: new "Clear Recently Played Channels" resets the PVR recently-
   played list. The write happens with the PVR manager paused so the running
   client cannot clobber it; it is a no-op with a notice when none are found.

2026.07.15.3
 - Clearing recently played channels now offers a Kodi restart. The home-screen
   widget reads the PVR "last played" list from the running manager's memory, so
   the cleared list only shows after Kodi reloads.

2026.07.15.4
 - Apple TV: the restart prompt now uses the honest "close and reopen Kodi"
   wording on tvOS. It was detected only as non-Android and mislabelled
   "Restart now?", but Kodi cannot self-restart on tvOS (Quit only closes).
   Fixes the wording for the new channel-clear restart offer and for every
   restore / Fresh Start prompt on Apple TV.

2026.07.16.0
 - Full backup now truly means full. The 2026.07.08.5 pvr.iptvsimple exclusion
   is reversed: IPTV settings (instance-settings, settings.xml, custom groups)
   are captured on BOTH Apple TV (from NSUserDefaults) and Fire TV, and a
   restore brings them back exactly as backed up. Stray instance files the
   archive does not carry are swept AFTER the extract (a canceled restore can
   never cost the box the config it already had), with every removal verified.
 - Restores briefly pause the IPTV client for the extract window and always
   re-enable it, so the live client can no longer overwrite the restored IPTV
   settings at shutdown.
 - Every backup embeds backup_manifest.json (what it holds, what it could not
   capture); restore verifies the archive against it. All backup / restore /
   wipe / Fresh Start reporting is now honest: partial results are reported as
   INCOMPLETE with the exact items, never "Complete", and an incomplete backup
   never rotates out the last good one.
 - Wipes on Apple TV clear BOTH storage layers (files and NSUserDefaults keys,
   verified against the store), and a one-time service migration purges stale
   hidden settings copies left by 2026.07.08.x-2026.07.13.x, materializing any
   key-only file to disk first so nothing is ever lost.
 - New Tools menu: "Purge stale tvOS keys" and "Verify backup archive".

2026.07.16.1
 - Verified live on real hardware (Fire TV + Apple TV): full backup captures
   IPTV on both OSes, restore brings it back exactly (strays swept, IPTV client
   paused and resumed), honest reporting confirmed. No functional change from
   2026.07.16.0 - this release records the hardware verification and the move to
   CI-driven publishing.

v2026.07.17.0
- Restore verifies its own work before reporting: a new "Verifying backup..." step triages
  everything the pre-restore wipe left behind against the backup (overwritten / harmless
  residue / dangerous stale keys) and auto-fixes the dangerous class with one silent fresh
  pass. The old mid-flight warning ("could not remove N items; leftovers may shadow...") is
  gone - it alarmed on states that were provably harmless and its modal could eat other
  prompts (this is exactly how a restored Apple TV came back in the stock skin).
- The user-facing vocabulary is now four owner-approved messages, with every count, path,
  and phase name demoted to the log: "Restore Complete" (the normal case), a Problem
  prompt with Try Again for genuine restore failures, a needs-attention line when a fix
  did not hold, and a silent post-restart self-check that only speaks on a real finding.
  "Restore INCOMPLETE" no longer appears for wipe imperfections - the restore phase and
  the wipe phase are reported as what they are. restore() now returns a structured summary
  (extracted/failed/unmapped/attention) so tooling keeps the full truth.
- Try Again re-runs the entire restore (fresh wipe included) from the already-staged
  backup, twice at most, then escalates instead of looping.
- The restored skin comes back reliably: lookandfeel.skin is no longer live-applied
  mid-restore (its keep-skin countdown was unanswerable behind our dialogs and reverted
  the skin). It is switched after all dialogs close and the confirmation is answered
  automatically, then verified.
- New restorecheck module (read-only verification: leftover triage, stale-key re-read,
  two-layer duplicate probe) shared by the restore and the boot self-check.

v2026.07.17.1
- Fix: the restored skin now reliably goes live on Apple TV after a restore.
  The v2026.07.17.0 cut answered Kodi's keep-this-skin dialog with SendClick(11)
  alone, which did not land on the Apple TV, so the box reverted to the stock
  skin (hardware-reproduced on atv2). _apply_boot_skin now falls back to remote
  navigation (proven to confirm on the device), VERIFIES the skin actually stuck
  after a settle, and retries several strategies before giving up loudly. New
  regression test reproduces the SendClick failure and proves the recovery.

v2026.07.17.2
- Fix: the post-restore verification no longer raises "needs attention" for a
  stale NSUserDefaults key the wipe could not clear when that key is at a path the
  backup does not restore - it shadows nothing restored, so it is cruft, logged not
  alarmed (hardware-repro atv2 2026-07-17: a clean restore falsely said "needs
  attention"). The genuine signal - a RESTORED file shadowed by a stale key - is a
  duplicate two-layer listing, which remains the attention case.

v2026.07.17.3
- Fix: the tvOS settings re-vector (a best-effort durability step after a restore) no
  longer raises "needs attention" when it misses some files. apply_guisettings has
  already applied the settings live (which persists on tvOS), so a re-vector miss is
  not a restore failure - it is logged. atv2 2026-07-17: a perfect restore kept saying
  "needs attention" because of this. Genuine functional failures (settings apply,
  IPTV pause/resume, stray-instance sweep, duplicate two-layer listings) still report.

v2026.07.17.4
- Fix: "needs attention" is now reserved for a genuinely broken restore. tvOS
  best-effort housekeeping that does not affect the restored box (a re-vector miss,
  an unclearable stale key not in the archive, an IPTV pause miss that did not
  materialize, a diagnostic probe that hiccuped) is LOGGED, not raised to the user.
  atv2 2026-07-17: a perfect restore kept saying "needs attention" over this noise.
  The signals that still report: settings-apply failure, IPTV re-enable failure,
  stray/duplicate IPTV instances, and a restored file shadowed by a stale key.
- The restore now publishes its verdict as a Home window property
  (ezm_restore_verdict / ezm_restore_findings) so it is readable over JSON-RPC for
  diagnostics, since this appliance's webserver /vfs refuses the log.

v2026.07.17.5
- Skin persistence rewrite: after a restore the restored skin is captured from the
  archive's guisettings BEFORE anything can clobber it, then written directly to disk
  and the tvOS NSUserDefaults layer as the LAST step - so the force-quit reopen loads
  it with no keep-this-skin dialog. Removes the flaky live-switch+SendClick confirm
  that reverted the Apple TV to stock ~1 in 4 restores. Includes a tvOS re-materialize
  guard so the write never blanks other settings.
- Silent-loss fix (merge/Dropbox path): a tvOS re-vector miss that could leave a stale
  key shadowing a restored file now reports needs-attention instead of a false
  "Complete". On the wipe path (keys already cleared) it stays harmless/log-only.
- A hard content failure surfacing on the auto-fix retry now shows the Problem wording,
  not the softer needs-attention.
- A wiped box is guaranteed to reach the restart prompt even on an unexpected error.
- Fresh Start: fixed an unpack bug that wiped the box then falsely reported the wipe
  did not run and skipped the restart. Now completes and restarts correctly.
- Set-up-this-box: the repo source rename to .T7B dedups by URL (no duplicate entries).
- Reviewed by three independent agents; the skin boot still owes one on-device run.

v2026.07.17.6
- Ended the boot-time war with skin.estuary7 1.0.66: the stale NSUserDefaults key
  purge now treats the skin's skinshortcuts *.DATA.xml keys (master profile AND
  per-profile) as skin-owned durability sidecars and never purges them, in either
  layer.
- Root-caused and fixed the purge's "failed" counting: post-delete verification
  re-read the NSUserDefaults backing plist, which cfprefsd flushes lazily on tvOS,
  so every successful key drop looked failed, the run-once marker never set, and
  the purge re-ran (and re-deleted the skin's keys) every boot. Verification now
  reads the LIVE layer via the xbmcvfs.listdir dup-count (a name listed twice has
  a surviving key; once means the key is gone).
- Only-copy safety tightened: a directory squatting on a key's path no longer
  counts as a disk twin (the key would have been the only real copy).
- Log viewer / pastebin uploader: crash-damaged logs (invalid UTF-8 written during
  a hard crash) no longer make both die silently; the bytes are replaced with
  markers and the log still displays/uploads. Cancel no longer picks the last log
  by accident, an empty log list says so instead of doing nothing, upload failures
  close the stuck busy dialog and are logged.
- Reviewed by two fix agents and two adversarial QA agents; tvOS hardware
  verification on atv2 recorded in verification/2026.07.17.6.json (android class
  waived: the changed storage path is tvOS-gated and never executes on Fire OS).
